在Android TEE 环境下添加与交互

在安卓 TEE（可信执行环境，如高通 QSEE、华为 iTrustee、联发科 TrustZone 等）中添加可信应用（TA）并实现与 Rich Execution Environment（REE，即安卓普通系统）的交互，核心是遵循 GlobalPlatform（GP）标准（TEE 领域通用规范），结合具体 TEE 厂商的开发工具链完成开发、编译、签名、部署和通信。以下是详细步骤和关键知识点：

一、核心概念铺垫

  1. TEE 与 REE：
    • REE：安卓系统运行的普通环境（Linux 内核 + 应用层），非可信；
    • TEE：独立于 REE 的隔离执行环境，硬件级安全（内存加密、权限隔离），仅运行可信应用（TA）。
  2. 关键组件：
    • TA（Trusted Application）：运行在 TEE 中的可信程序，处理敏感逻辑（如密钥生成、加密、生物认证）；
    • CA（Client Application）：运行在 REE 中的普通应用，通过 TEE Client API 与 TA 通信；
    • TEE Manager：TEE 内核组件，负责 TA 的加载、卸载、权限管理；
    • GP 标准接口：定义了 CA-TA 通信、TA 生命周期管理的统一规范（如 TEE Client API、TA 接口函数）。

二、前提条件

  1. 硬件支持：安卓设备需搭载支持 TEE 的芯片（如高通骁龙、华为麒麟、联发科天玑），且 TEE 已激活（多数量产设备默认启用）；
  2. 开发工具链：
    • 厂商提供的 TEE SDK（如高通 QSEE SDK、华为 TEE SDK、联发科 TrustZone SDK），或开源 TEE 工具链（如 OP-TEE）；
    • 交叉编译器（针对 TEE 的 CPU 架构，如 aarch64-linux-android-）；
    • 签名工具（TA 必须通过 TEE 信任的密钥签名才能加载，厂商通常提供签名脚本或工具）；
  3. 系统权限：
    • 部署 TA 需设备 root 权限（开发/测试阶段），或通过厂商的系统签名（量产阶段）；
    • 部分 TEE 支持“测试签名”模式，方便开发调试。

三、步骤 1：开发可信应用（TA）

TA 的开发需遵循 GP 规范，核心是实现 GP 定义的生命周期接口和自定义业务接口。以下以 OP-TEE（开源 TEE，适合学习） 为例（厂商 TEE 开发流程类似，仅工具链和签名不同）。

1.1 TA 的目录结构（OP-TEE 示例）

ta/
├── include/          # 头文件（GP 接口、自定义结构体）
│   └── ta_sample.h
├── src/
│   └── ta_sample.c   # TA 核心逻辑
├── Makefile          # 编译脚本（指定编译器、签名密钥）
└── ta_dev_kit/       # TEE 开发套件（厂商提供，含 GP 接口定义）

1.2 实现 TA 核心接口（GP 规范强制要求）

TA 必须实现以下 4 个生命周期接口（由 GP 定义），以及自定义的业务接口：

#include <tee_internal_api.h>
#include <tee_internal_api_extensions.h>

// TA 唯一标识符（UUID），CA 需通过此 UUID 找到 TA
#define TA_SAMPLE_UUID \
    {0x12345678, 0x1234, 0x5678, {0x12, 0x34, 0x56, 0x78, 0x9a, 0xbc, 0xde, 0xf0}}

// 1. TA 初始化（加载时调用）
TEE_Result TA_CreateEntryPoint(void) {
    IMSG("TA 创建成功");
    return TEE_SUCCESS;
}

// 2. TA 销毁（卸载时调用）
void TA_DestroyEntryPoint(void) {
    IMSG("TA 销毁成功");
}

// 3. 打开会话（CA 与 TA 建立连接时调用）
TEE_Result TA_OpenSessionEntryPoint(
    uint32_t param_types,
    TEE_Param params[4],
    void **session_ctx) {
    IMSG("会话打开成功");
    *session_ctx = NULL; // 会话上下文（可存储会话相关数据）
    return TEE_SUCCESS;
}

// 4. 关闭会话（CA 与 TA 断开连接时调用）
void TA_CloseSessionEntryPoint(void *session_ctx) {
    IMSG("会话关闭成功");
}

// 5. 自定义业务接口（CA 调用 TA 的核心逻辑）
// cmd：命令码（CA 与 TA 约定，区分不同操作）
// param_types：参数类型（输入/输出/双向）
// params：参数数据（最多 4 个，支持整数、内存块等）
TEE_Result TA_InvokeCommandEntryPoint(
    void *session_ctx,
    uint32_t cmd,
    uint32_t param_types,
    TEE_Param params[4]) {
    switch (cmd) {
        case 0x01: { // 示例：加解密命令
            // 从 params 中获取 REE 传入的数据（如明文、密钥）
            uint8_t *plaintext = params[0].memref.buffer;
            uint32_t plaintext_len = params[0].memref.size;
            uint8_t *key = params[1].memref.buffer;
            uint32_t key_len = params[1].memref.size;
            
            // 敏感操作：在 TEE 中执行加密（示例用简单异或，实际用 TEE 内置加密 API）
            for (int i = 0; i < plaintext_len; i++) {
                plaintext[i] ^= key[i % key_len];
            }
            
            // 结果通过 params 返回给 REE（params[0] 设为输出类型）
            return TEE_SUCCESS;
        }
        default:
            return TEE_ERROR_BAD_PARAMETERS;
    }
}

1.3 编译 TA

通过厂商提供的 Makefile 编译，生成 TA 二进制文件（通常后缀为 .ta）：

# OP-TEE 编译命令（需配置交叉编译器路径）
make -C ta CROSS_COMPILE=aarch64-linux-android-

编译后生成的 sample.ta 包含：TA 代码、GP 元数据（UUID、权限）、签名信息。

四、步骤 2：开发客户端应用（CA）

CA 是运行在安卓 REE 中的普通应用（APK），通过 TEE Client API（GP 标准）与 TA 交互。核心流程：打开 TEE 设备 → 连接 TA → 发送命令 → 接收结果 → 断开连接。

2.1 依赖 TEE Client 库

安卓系统已内置 TEE Client 库（libteec.so），CA 需通过 JNI 调用（Java 无法直接访问 native 库），或使用厂商提供的 Java 封装 SDK。

2.1.1 JNI 层实现（核心通信逻辑）

#include <jni.h>
#include <teec.h>
#include <android/log.h>

#define LOG_TAG "TEE_CA"
#define LOGD(...) __android_log_print(ANDROID_LOG_DEBUG, LOG_TAG, __VA_ARGS__)

// TA 的 UUID（与 TA 中定义一致）
static const TEEC_UUID TA_SAMPLE_UUID = {
    0x12345678, 0x1234, 0x5678,
    {0x12, 0x34, 0x56, 0x78, 0x9a, 0xbc, 0xde, 0xf0}
};

// 命令码（与 TA 约定）
#define CMD_ENCRYPT 0x01

// JNI 方法：调用 TA 执行加密
JNIEXPORT jbyteArray JNICALL
Java_com_example_teeclient_TeeManager_encrypt(JNIEnv *env, jobject thiz, jbyteArray plaintext, jbyteArray key) {
    TEEC_Context ctx;
    TEEC_Session sess;
    TEEC_Operation op;
    TEEC_Result res;

    // 1. 初始化 TEE 上下文（连接 TEE 驱动）
    res = TEEC_InitializeContext(NULL, &ctx);
    if (res != TEEC_SUCCESS) {
        LOGD("初始化 TEE 失败：%d", res);
        return NULL;
    }

    // 2. 打开与 TA 的会话（通过 UUID 找到 TA）
    res = TEEC_OpenSession(&ctx, &sess, &TA_SAMPLE_UUID,
                          TEEC_LOGIN_PUBLIC, NULL, NULL, NULL);
    if (res != TEEC_SUCCESS) {
        LOGD("打开 TA 会话失败：%d", res);
        TEEC_FinalizeContext(&ctx);
        return NULL;
    }

    // 3. 准备参数（传递给 TA）
    jbyte *plaintext_buf = (*env)->GetByteArrayElements(env, plaintext, NULL);
    jsize plaintext_len = (*env)->GetArrayLength(env, plaintext);
    jbyte *key_buf = (*env)->GetByteArrayElements(env, key, NULL);
    jsize key_len = (*env)->GetArrayLength(env, key);

    memset(&op, 0, sizeof(op));
    op.paramTypes = TEEC_PARAM_TYPES(
        TEEC_MEMREF_INOUT,  // 参数 0：输入输出（明文/密文）
        TEEC_MEMREF_INPUT,  // 参数 1：输入（密钥）
        TEEC_NONE, TEEC_NONE
    );
    op.params[0].memref.buffer = plaintext_buf;
    op.params[0].memref.size = plaintext_len;
    op.params[1].memref.buffer = key_buf;
    op.params[1].memref.size = key_len;

    // 4. 调用 TA 的命令（触发 TA_InvokeCommandEntryPoint）
    res = TEEC_InvokeCommand(&sess, CMD_ENCRYPT, &op, NULL);
    if (res != TEEC_SUCCESS) {
        LOGD("调用 TA 命令失败：%d", res);
        (*env)->ReleaseByteArrayElements(env, plaintext, plaintext_buf, JNI_ABORT);
        (*env)->ReleaseByteArrayElements(env, key, key_buf, JNI_ABORT);
        TEEC_CloseSession(&sess);
        TEEC_FinalizeContext(&ctx);
        return NULL;
    }

    // 5. 处理结果（TA 已修改 plaintext_buf 为密文）
    jbyteArray ciphertext = (*env)->NewByteArray(env, plaintext_len);
    (*env)->SetByteArrayRegion(env, ciphertext, 0, plaintext_len, plaintext_buf);

    // 6. 释放资源
    (*env)->ReleaseByteArrayElements(env, plaintext, plaintext_buf, JNI_ABORT);
    (*env)->ReleaseByteArrayElements(env, key, key_buf, JNI_ABORT);
    TEEC_CloseSession(&sess);
    TEEC_FinalizeContext(&ctx);

    return ciphertext;
}

2.1.2 Java 层封装（供安卓应用调用）

package com.example.teeclient;

public class TeeManager {
    // 加载 JNI 库
    static {
        System.loadLibrary("tee_ca");
    }

    // 调用 TEE 加密的接口
    public native byte[] encrypt(byte[] plaintext, byte[] key);
}

2.1.3 安卓应用调用

// 主线程或子线程中调用
TeeManager teeManager = new TeeManager();
byte[] plaintext = "hello tee".getBytes();
byte[] key = "123456".getBytes();
byte[] ciphertext = teeManager.encrypt(plaintext, key);
Log.d("TEE_DEMO", "加密结果：" + new String(ciphertext));

五、步骤 3：TA 签名与部署

TA 必须通过 TEE 信任的密钥签名才能被加载（防止恶意 TA 注入），部署需将签名后的 TA 放入设备的 TEE 存储目录。

3.1 TA 签名

  • 开发阶段：使用厂商提供的“测试密钥”签名（如 OP-TEE 的 default_ta.key），Makefile 中配置签名密钥路径即可自动签名；
  • 量产阶段：使用厂商颁发的正式密钥签名（确保 TA 来源可信）。

3.2 部署 TA 到设备

  1. 设备 root 后，通过 adb 将 .ta 文件推送到 TEE 的 TA 存储目录（不同厂商路径不同）：

    • OP-TEE：/vendor/lib/optee_armtz/
    • 高通 QSEE：/firmware/image/（需打包为镜像）
    • 华为 iTrustee：/vendor/etc/tee/

  2. 重启设备或通过 TEE Manager 加载 TA：

    # 示例：推送到 OP-TEE 目录
adb push sample.ta /vendor/lib/optee_armtz/
adb shell chmod 644 /vendor/lib/optee_armtz/sample.ta

六、步骤 4：CA-TA 交互测试

  1. 编译 CA 为 APK 并安装到设备；
  2. 运行 APK，调用 encrypt 方法；
  3. 查看日志（通过 adb logcat）：
    • 若 TA 加载成功：日志显示 TA 创建成功、会话打开成功；
    • 若加密成功：日志显示 调用 TA 命令失败：0（TEEC_SUCCESS=0），并输出加密结果。

七、关键注意事项

  1. 参数传递限制：
    • CA 与 TA 之间的参数通过共享内存传递（TEE 会验证内存权限），最多支持 4 个参数（TEE_Param）；
    • 敏感数据（如密钥）需在 CA 中加密后传递，或直接在 TA 中生成（避免明文暴露在 REE）。
  2. TA 权限控制：
    • TA 的 UUID 是唯一标识，需确保不与其他 TA 冲突；
    • 可通过 GP 规范的“访问控制列表（ACL）”限制 CA 调用权限。
  3. 厂商差异：
    • 不同 TEE 厂商的 SDK、TA 存储路径、签名方式不同（需参考厂商文档）；
    • 部分厂商不开放 TEE 开发权限（仅提供预制 TA 的调用接口）。
  4. 调试工具：
    • OP-TEE 提供 tee-supplicant 工具用于调试；
    • 可通过 adb shell dmesg 查看 TEE 内核日志。

八、开源工具链推荐（学习用）

  • OP-TEE：开源 TEE 解决方案，支持多种芯片，提供完整的 TA/CA 开发工具链（适合入门学习）；
  • TrustZone Tools：ARM 提供的 TrustZone 开发工具，含调试器、性能分析工具。

总结

安卓 TEE 中添加 TA 并实现交互的核心流程是：遵循 GP 规范开发 TA（实现生命周期接口+业务接口）→ 开发 CA（通过 TEE Client API 通信）→ 签名部署 TA → 测试交互。关键是理解 TEE 与 REE 的隔离特性，确保敏感逻辑在 TA 中执行，数据传输过程的安全性。实际开发需结合具体厂商的 SDK 和文档，解决签名、部署等厂商相关问题。
